# 🧩 후기 문제

## [곱셈 ](https://www.acmicpc.net/problem/1629)

### 💡 아이디어 / 접근법
- 처음에는 나머지가 반복되는 주기를 찾아 배열에 저장하는 방식으로 접근했다.(이러면 분할정복이 아니긴 하지만 사이클을 찾으면 분할정복 굳이 필요한가? 싶어서 시도했었다) 하지만 나누는 수 C가 최대 21억이라 최악의 경우 21억 번을 돌아야 해서 메모리와 시간 초과가 발생할 수밖에 없었다.
- 다음으로는 단순 for문을 돌며 매번 % C를 해주는 방식을 시도했다. 오버플로우는 막을 수 있었지만, 곱하는 횟수 B가 21억이면 시간 제한(0.5초)을 절대 통과할 수 없는 O(N) 방식이었다.
- 최종적인 해결책은 분할 정복을 이용해 지수를 반으로 쪼개는 것이었다 거듭제곱의 성질을 이용해 연산 횟수를 21억 번에서 약 30번대(O(log N))로 획기적으로 줄였다.

핵심 개념:
재귀의 함정: 
1. 처음엔 분할정복이니 각 케이스마다 재귀를 계속 깊게 들어가면서 나누려고 divide(n/2)를 두번씩 했었다. 짝수일 때 divide(n/2) * divide(n/2)처럼 함수를 두 번 호출하면 가지치기가 되어 결국 시간 복잡도가 다시 O(N)으로 돌아가버린다. 이를 해결하기 위해 half = divide(n/2)로 값을 한 번만 구해서 캐싱한 뒤 half * half를 하는 방식으로 뜯어고쳤다.
2. 홀수 처리: 지수가 11인 경우 5와 6으로 나누는 대신, 5짜리(half)를 두 번 곱하고 마지막에 원래 숫자(A)를 한 번 더 곱해주는 방식으로 깔끔하게 해결했다.
3. 오버플로우 방지: half * half를 하는 순간에도 long 범위를 터뜨릴 수 있기 때문에, 매번 곱셈이 일어날 때마다 잊지 않고 % C 연산을 끼워 넣었다.

### 🤔 후기
- A를 B번 곱하라고 해서 진짜 B번 곱하면 안 된다는 걸 뼈저리게 느낀 문제였다. 지수를 반으로 쪼갠다는 수학적 발상 자체도 신기했다.
- 재귀 함수를 짤 때 단순히 로직만 맞추는 게 아니라, 똑같은 연산을 불필요하게 두 번 호출하지 않도록 캐싱하는 것이 성능에 얼마나 치명적인 영향을 주는지 확실하게 배울 수 있었다.

## [행렬 곱셈](https://www.acmicpc.net/problem/2740)

### 💡 아이디어 / 접근법

- 다행히 시간초과가 발생하지 않고 문제풀이 성공

### 🤔 후기

- 행열 계산하는법을 먼저 찾아보고 그려서 풀어본다음 규칙 찾아서 풀이 성공했다..
